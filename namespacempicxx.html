Enumeration Type Documentation

enum mpicxx::thread_support
strong

Enum class for the different levels of thread support provided by MPI.

The values are monotonic: single < funneled < serialized < multiple.

Enumerator
single 

only one thread will execute

funneled 

the process may be multi-threaded, but the application must ensure that only the main thread makes MPI calls (see mpicxx::is_main_thread())

serialized 

the process may be multi-threaded, and multiple threads may make MPI calls, but only one at a time

multiple 

multiple threads may make MPI calls, with no restrictions

void mpicxx::finalize ( )
inline

Clean up the MPI state.

If a MPI program terminates normally (i.e., not due to a call to mpicxx::abort() or an unrecoverable error) then each process must call mpicxx::finalize() before it exits. Before an MPI process invokes mpicxx::finalize(), the process must perform all MPI calls needed to complete its involvement in MPI communications.

Precondition assertions
If the MPI environment has already been finalized.

int mpicxx::atfinalize ( detail::atfinalize_callback_t  func)
inline

Registers the callback function func (of the type void (*)void) to be called directly before MPI_Finalize.

Calls all registered functions in reverse order in which they were set. This happens before any other parts of MPI are affected, i.e. mpicxx::finalized() will return false in any of these callback functions.

The maximum number of registrable callback functions can be set via the CMake configuration flag MPICXX_MAX_NUMBER_OF_ATFINALIZE_CALLBACKS (default: 32).

template<typename FuncPtr , typename... Args>
requires detail::is_main_pointer<FuncPtr, Args...> int mpicxx::main ( FuncPtr  func,
Args &&...  args 
)
inline

Correctly setup and teardown the MPI environment while executing the code given by func.

This function performs the following tasks in the given order:

  1. call mpicxx::init()
  2. invoke the function represented by func (forwarding all additional parameters)
  3. call mpicxx::finalize()

Example:

#include <mpicxx/startup/init.hpp>
#include <mpicxx/startup/finalize.hpp>
int main() {
mpicxx::init(); // don't forget the initialization call (exactly once)!
{ // brackets needed or mpicxx objects would get destroyed after the finalization call which is not permitted
// user code
}
mpicxx::finalize(); // don't forget the finalization call (exactly once)!
return 0;
}

is the same as:

#include <mpicxx/startup/mpicxx_main.hpp>
int mpicxx_main() {
// no need to call a initialization function!
// user code
// no need to call a finalization function!
return 0;
}
int main() {
return mpicxx::main(&mpicxx_main); // can't make any mistake related to initialization or finalization!
}

Correctly setup and teardown the MPI environment with the required level of thread support while executing the code given by func. If the required level of thread support couldn't be satisfied, the function returns immediately with return code EXIT_FAILURE.

This function performs the following tasks in the given order:

  1. call mpicxx::init(const thread_support)
  2. invoke the function represented by func (forwarding all additional parameters)
  3. call mpicxx::finalize()

Example:

#include <mpicxx/startup/init.hpp>
#include <mpicxx/startup/finalize.hpp>
int main() {
try {
mpicxx::init(mpicxx::thread_support::MULTIPLE); // don't forget the initialization call (exactly once)!
// user code
} catch(const mpicxx::thread_support_not_satisfied& e) {
std::cout << e.what() << std::endl;
}
mpicxx::finalize(); // don't forget the finalization call even in case of an exception (exactly once)!
return 0;
}

is nearly the same as (except for the return value):

#include <mpicxx/startup/mpicxx_main.hpp>
int mpicxx_main() {
// no need to call a initialization function!
// no need to catch an exception; if this function gets called, the required level of thread support could be satisfied
// user code
// no need to call a finalization function!
return 0;
}
int main() {
return mpicxx::main(&mpicxx_main, mpicxx::thread_support::MULTIPLE); // can't make any mistake related to initialization or finalization!
}
